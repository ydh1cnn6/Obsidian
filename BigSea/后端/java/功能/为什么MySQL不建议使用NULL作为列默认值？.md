
MySQL不建议用NULL作为列默认值，核心原因是NULL不参与索引、会导致查询逻辑复杂、占用额外存储空间，还可能引发统计偏差和运算错误。
 
关键问题拆解
 
1. 索引失效风险：B树索引不会存储NULL值， WHERE 列 IS NULL 无法利用索引，只能全表扫描，大数据量下查询变慢；复合索引中若某列默认NULL，会导致索引利用率大幅下降。
2. 逻辑判断复杂：NULL≠任何值（包括自身），需用 IS NULL/IS NOT NULL 判断，而非 =/- ，容易写出bug（如 WHERE 列 = NULL 永远返回false）；聚合函数（如COUNT、SUM）会忽略NULL，导致统计结果不符合预期。
3. 存储与性能损耗：InnoDB中，NULL值需额外存储“NULL标记”（约1字节/列），比存储空字符串（''）等非NULL默认值更占用空间；大量NULL列会增加数据页碎片，降低IO效率。
4. 兼容性与迁移问题：部分数据库（如PostgreSQL）对NULL的处理逻辑不同，且NULL在数据迁移、ETL过程中易引发格式转换错误，增加维护成本。
 
替代方案
 
优先用有意义的默认值：如字符串用''、数字用0/1、日期用 CURRENT_TIMESTAMP ，既避免NULL的弊端，又让业务逻辑更清晰。
